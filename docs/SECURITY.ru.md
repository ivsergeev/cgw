# Архитектура безопасности CorpGateway

[English version](../SECURITY.md)

Этот документ описывает механизмы безопасности CorpGateway, угрозы от которых они защищают, и их ограничения.

---

## Обзор

CorpGateway использует четырёхуровневую модель безопасности:

```
┌─────────────────────────────────────────────────────────┐
│  Уровень 1: HTTP API (cgw_mcp)                         │
│  Уровень 2: WebSocket (cgw_mcp ↔ расширение)           │
│  Уровень 3: Chrome Extension                            │
│  Уровень 4: Данные                                      │
└─────────────────────────────────────────────────────────┘
```

---

## Уровень 1: HTTP API (cgw_mcp)

### Аутентификация по токену

Все HTTP-запросы к `/mcp` и `/health` требуют Bearer-токен:

```
Authorization: Bearer <token>
```

Токен генерируется автоматически (128-бит hex) при первом запуске сервера и хранится в `~/.corpgateway/cgw_mcp.json`.

### Timing-safe сравнение

Сравнение токенов использует `crypto.timingSafeEqual()` для защиты от timing-атак. При разной длине токенов выполняется фиктивное сравнение для поддержания постоянного времени ответа.

### Rate limiting

Неудачные попытки аутентификации отслеживаются по IP-адресу:
- **Лимит:** 10 неудач за 1-минутное окно
- **Ответ:** HTTP 429 (Too Many Requests)
- **Очистка:** устаревшие записи удаляются каждые 5 минут

Применяется как к HTTP-эндпоинтам, так и к WebSocket-подключениям.

### Политика CORS

Разрешены только localhost-origin:

```
Access-Control-Allow-Origin: http://localhost[:port] | http://127.0.0.1[:port]
```

Запросы с любых других origin не получают CORS-заголовки — браузер их блокирует.

### Whitelist методов

Только эти JSON-RPC методы пересылаются в расширение:
- `initialize`, `ping`, `tools/list`, `tools/call`, `notifications/initialized`

Все остальные отклоняются с кодом ошибки `-32601`.

### Лимиты размера сообщений

- **HTTP тело:** максимум 1 МБ (запрос уничтожается при превышении)
- **WebSocket:** максимум 1 МБ (`maxPayload` в WebSocketServer)

### Таймауты запросов

- **HTTP:** 60-секундный таймаут на запрос и ответ
- **Ответ расширения:** 30-секундный таймаут на каждый перенаправленный запрос
- **Лимит параллельных:** максимум 100 ожидающих запросов

### Коды ошибок

Структурированные коды помогают агентам понять причину сбоя без раскрытия внутренней архитектуры:

| Код | Значение |
|-----|----------|
| `-32001` | Расширение не подключено |
| `-32002` | Таймаут запроса |
| `-32003` | Сервер перегружен |
| `-32004` | Соединение с расширением потеряно |

### Привязка к localhost

Сервер слушает только на `127.0.0.1` — не доступен из сети.

## Уровень 2: WebSocket (Mutual Authentication)

WebSocket-соединение между cgw_mcp и расширением использует трёхшаговый протокол взаимной аутентификации на основе HMAC-SHA256.

### Протокол

```
Extension → Server:  WS connect с ?token=<extensionToken>
Server:              Проверяет extensionToken (односторонняя аутентификация)

Server → Extension:  { type: "challenge",
                       challenge: <случайные 32 байта>,
                       serverProof: HMAC-SHA256(extensionToken, "server:" + challenge) }
Extension:           Проверяет serverProof ← доказывает, что сервер знает extensionToken

Extension → Server:  { type: "challenge-response",
                       hmac: HMAC-SHA256(extensionToken, "client:" + challenge) }
Server:              Проверяет hmac ← доказывает, что расширение знает extensionToken

Server → Extension:  { type: "authenticated" }
```

### От чего защищает

- **Поддельный сервер:** атакующий не может создать валидный `serverProof` без знания `extensionToken`. Расширение проверяет proof и закрывает соединение при несовпадении.
- **Поддельное расширение:** атакующий не может создать валидный `clientProof` без `extensionToken`. Сервер закрывает соединение при несовпадении.
- **Replay-атаки:** каждый challenge — свежее 32-байтное случайное значение. Префиксы `"server:"` / `"client:"` предотвращают использование одного HMAC для обеих ролей.

### Защита от timing-атак

- Сервер использует `crypto.timingSafeEqual()` для сравнения HMAC
- Расширение использует Web Crypto API (`crypto.subtle`), обеспечивающий constant-time операции
- Таймаут аутентификации: 5 секунд (соединение закрывается при превышении)

### Обработка сообщений

Никакие сообщения не обрабатываются до завершения взаимной аутентификации. Расширение отслеживает состояние флагами `serverAuthenticated` и `challengeSent` — сообщения вне последовательности вызывают немедленное отключение.

## Уровень 3: Chrome Extension

### Валидация отправителя

Обработчик сообщений расширения проверяет отправителя каждого сообщения:

- **`getStatus`** — разрешено от любого отправителя (только чтение, используется overlay)
- **`connect`, `disconnect`, `mcpRequest`, `configUpdated`, `skillsChanged`** — только со страниц расширения (popup, options). Проверяется `sender.id === chrome.runtime.id` и `sender.url.startsWith(chrome.runtime.getURL(''))`.

Сообщения от веб-страниц, content-скриптов на произвольных сайтах или других расширений отклоняются.

### Защита от SSRF

Перед выполнением каждого скила финальный URL и `fetchOrigin` проверяются:

**Заблокированы:**
- Не-HTTP протоколы (`file:`, `ftp:`, `javascript:`, `data:` и др.)
- `localhost`, `::1`
- Приватные IP: `10.x`, `172.16-31.x`, `192.168.x`, `169.254.x`
- IPv6 приватные: `fc00::/7`, `fd00::/8`, `fe80::/10`
- `0.0.0.0/8`

Это предотвращает использование скилов для сканирования или доступа к внутренним сетям.

### Защита от инъекций в шаблонах

Подстановка параметров в телах запросов и заголовках санитизируется:

- **Заголовки:** значения с `\r` или `\n` отклоняются (защита от CRLF-инъекций)
- **JSON тело:** значения экранируются через `JSON.stringify()`. После подстановки результат проверяется `JSON.parse()` на валидность JSON
- **Целые/дробные числа:** проверяются `Number.isFinite()` (предотвращает NaN/Infinity)
- **Булевы:** строгий whitelist (`true`, `1`, `yes` → true; всё остальное → false)

### Валидация импорта скилов

Импортируемые файлы пресетов проверяются перед сохранением:

- Протокол URL: только `http:` или `https:`
- HTTP метод: один из `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`
- Максимум 50 параметров на скил
- Максимум 20 заголовков на скил
- Типы параметров: `String`, `Integer`, `Float`, `Boolean`, `Date`
- Дубликаты скилов/групп пропускаются (case-insensitive)

### Подтверждение операций (OTP)

Скилы с `confirm: true` требуют одноразовый код подтверждения перед выполнением.

**Процесс:**
1. Агент вызывает `cgw_invoke(skill, params)`
2. Расширение генерирует 4-значный код (криптографически случайный)
3. Код показывается пользователю через системное уведомление
4. Расширение возвращает агенту "требуется подтверждение"
5. Агент просит пользователя назвать код
6. Агент вызывает `cgw_invoke(skill, params, confirmCode)`
7. Расширение проверяет код и выполняет операцию

**Свойства:**
- Код: 4 случайные цифры (0000–9999), `crypto.getRandomValues()`
- Действителен 60 секунд, затем истекает
- Одноразовый — потребляется после проверки
- Привязан к конкретному скилу + параметрам (SHA-256 хэш)
- Код существует только в системном уведомлении — агент его не видит
- `confirmCode` исключается из хэширования параметров

**От чего защищает:**
- Prompt injection, вызывающий непреднамеренные операции записи
- Автономное выполнение деструктивных действий агентом без ведома пользователя

### Уведомление при ошибке авторизации

Когда скил возвращает 401 после автоматической попытки обновления авторизации, расширение показывает системное уведомление с просьбой повторно войти в систему.

## Уровень 4: Защита данных

### Права доступа к файлу конфигурации

`~/.corpgateway/cgw_mcp.json` содержит оба токена:

- **Директория:** создаётся с правами `0o700` (только владелец)
- **Файл:** создаётся с правами `0o600` (чтение/запись только владельцу)
- **Контроль:** права проверяются и исправляются при каждом запуске сервера (Unix/macOS)

### Маскирование токенов в логах

Токены никогда не записываются в лог-файлы полностью. Показываются только первые 4 и последние 4 символа:

```
Agent token:     a1b2...f8e7
Extension token: c3d4...h2i1
```

### Аудит-лог

Все вызовы скилов записываются в `chrome.storage.session`:

- Последние 100 записей
- Каждая запись: имя скила, имена параметров (не значения), ошибки, длительность, метка времени
- Требовалось ли подтверждение
- Очищается при перезапуске расширения (session storage)
- Доступен через инструмент `cgw_audit`

### Chrome Storage

Скилы, группы и конфигурация расширения хранятся в `chrome.storage.local`, который шифруется Chrome и доступен только расширению.

## Сводка по безопасности

| Область | Защита |
|---------|--------|
| Аутентификация по токену | Timing-safe сравнение + rate limiting (10/мин на IP) |
| Идентификация сервера | Mutual HMAC-SHA256 challenge-response на WebSocket |
| Идентификация расширения | Extension token + HMAC-верификация |
| Межсайтовые запросы | CORS ограничен localhost + Bearer token |
| Методы API | Whitelist разрешённых JSON-RPC методов |
| Размер сообщений | Лимит 1 МБ на HTTP и WebSocket |
| Стабильность соединения | 60-секундные таймауты + макс. 100 параллельных запросов |
| Доступ к внутренней сети | Валидация URL блокирует приватные IP и не-HTTP протоколы |
| Инъекции в запросах | CRLF-проверка в заголовках + JSON-валидация в теле |
| Деструктивные операции | Двойное: `cgw_invoke_confirmed` (нативный prompt) + OTP fallback |
| Импорт скилов | Валидация протокола, метода, типов и количества параметров |
| Источник сообщений | Валидация отправителя — только страницы расширения могут слать команды |
| Файл конфигурации | Ограниченные права доступа (0600 на Unix) |
| Безопасность логов | Токены маскируются до первых/последних 4 символов |
| Аудит | Последние 100 вызовов с метками времени |
| Сетевая изоляция | Сервер слушает только на 127.0.0.1 |

## Рекомендации

- Используйте **отдельный профиль Chrome** для CorpGateway для изоляции корпоративных сессий
- **Включайте только нужные** группы скилов — отключайте неиспользуемые
- **Включайте подтверждение** на скилах, выполняющих операции записи
- Держите cgw_mcp и расширение **в актуальном состоянии**
